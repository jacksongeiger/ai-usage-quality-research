# Survey: Measuring Agent/Automation/Platform Engineers (Non-PR Output)

Sub-investigator area: agent-platform telemetry metrics, platform-engineering value measurement, IDP ROI, automation impact measurement, non-PR knowledge-work measurement. Feeds the brief's hardest requirement: agent-platform engineers score near-zero today (brief hard limit: "No per-user OUTPUT signal for agent-platform / non-PR work").

**Headline honesty statement:** There is NO published, validated methodology for scoring an *individual engineer* by the value of the agents/automation/platforms they build. Every credible framework found measures one of: (a) the *agent's* runtime performance (observability/evals), (b) the *platform's* org- or team-level ROI, or (c) treats agents as extensions of a *team* and punts on individual attribution. The per-individual agent-builder score the brief asks for must be synthesized from transferable patterns; the strongest transferable pattern, consistently across four independent traditions (Backstage, CNCF, software-reuse research, agent KPI frameworks), is **adoption/reuse by other people, especially repeat use** — which requires one NEW signal (agent ownership + caller identity per run) that the platform almost certainly already logs but the warehouse does not expose (inference).

## 1. What it measures & theory

### 1a. Agent-platform runtime telemetry (what actually exists per-user)
- **OpenTelemetry GenAI semantic conventions** (the emerging industry standard; still labeled "Development" as of early 2026, now maintained in its own repo) standardize spans for agent lifecycle (`create_agent`, `invoke_agent`), tool execution (`execute_tool`), plus metrics `gen_ai.client.operation.duration` and `gen_ai.client.token.usage`. Theory: make agent behavior traceable/comparable across frameworks. **Critically: the spec captures errors, latency, tokens, and call structure — it has NO notion of task success or output value.** (Span names cross-reported by multiple independent summaries; spec page itself redirects to the repo — see Sources.)
- **Langfuse / LangSmith-class observability**: traces, nested observations, sessions, users, tags, costs. "Success" exists only as a **score** — a builder-defined evaluation attached to traces, sourced from human annotation, user feedback, LLM-as-judge, or custom API. Langfuse docs are explicit that there is no built-in notion of run success; the builder defines what success means. Theory: evaluation is application-specific; the platform provides plumbing only.
- Implication for our HAVE list: an agent platform that logs "spend, tokens, sessions, active days, run counts but NOT artifacts" is exactly the industry-standard telemetry floor. Nothing in the standard stack gives per-user *output value* for free.

### 1b. Production-agent KPI frameworks (org-level)
- **Google Cloud, "The KPIs that actually matter for production AI agents" (2026-02-26)** — most concrete framework found. Three pillars:
  1. *Reliability & operational efficiency*: tool-selection accuracy, argument-hallucination rate, plan adherence, consistency score (audited by a "critic agent" over execution logs); **cost per successful task** ("$0.10/run with 50% failure = $0.20 actual cost per success" — pair cost with success, never tokens in isolation); planning efficiency; end-to-end latency (catches "analysis paralysis" loops).
  2. *Adoption & usage*: for user-invoked agents — active users, invocation rate per active user, session depth, retention of generated text; for background agents — acceptance rate, **implicit rejection rate (undo/revert tracking, "stronger signal than explicit thumbs-down")**, handoff-verification latency, output friction (accepted as-is / edited / reverted).
  3. *Business value*: time-to-value acceleration, OpEx reduction (manual steps eliminated), new capabilities unlocked, revenue acceleration.
  - Theory: trust first (reliability), then friction (adoption), then ROI. Everything measures the *agent or the org* — the builder is never the unit of analysis.
- **DX (getdx.com), "Measuring AI code assistants and agents"**: explicitly recommends treating agents "as extensions of the developers and teams that oversee their work"; team metrics should "include both human-authored pull requests and those authored by agents operating under that team's direction," analogous to how managers are measured via their teams. Openly defers agent-specific metric standardization: "agentic tooling is still in its early stages... measurement strategies and working models must evolve." Note: "human-equivalent agent hours" appears in this repo's prior SOURCES.md entry [7] for DX, but the research page fetched 2026-07-08 does not define it — **unverified; do not build on it**.
- Customer-service agent KPI literature (resolution rate vs. deflection rate, containment, cost-per-resolution) is mature but measures a *product*, not an engineer; ROI figures in that space ("3–6x ROI in year one") are vendor marketing — noted only to exclude.

### 1c. Platform-engineering / IDP value measurement
- **Spotify Backstage** (the canonical public IDP case, via InfoQ 2023; Spotify's own blog post now 404s): north-star was **onboarding time = time for a new developer to merge their 10th PR** (reported to fall from ~60 to <20 days); plus engagement (unique users, visit frequency), **plugin-contribution ecosystem metrics — number of teams contributing ≥1 plugin, total plugins, % of contributions from outside the central platform team (~85% reported)** — and correlational claims (frequent Backstage users 2.3x more active on GitHub; higher retention). Theory: platform value = adoption + ecosystem contribution + delta on a company-level flow metric. **All team/org-level; correlation-vs-causation not addressed** (heavy users being more active on GitHub is classic selection effect — inference).
- **CNCF Platform Engineering Maturity Model** (TAG App Delivery whitepaper): dedicated "Measurement" dimension with 4 maturity levels — L1 ad hoc/anecdotal → L2 consistent collection of **user adoption rates and user time saved** → L3 outcome-first metric selection (build times, lead time, reliability) → L4 mixed quant+qual, democratized, with an explicit **Goodhart's Law warning** ("when a measure becomes a target, it ceases to be a good measure") and continuous re-evaluation of metrics for gaming. Theory: measurement maturity itself is the capability; named metrics stay org-level.
- **DORA 2024** (platform-engineering chapter): platforms raise individual/team/org productivity (~5% via developer independence) but were associated with an **~8% throughput decrease and ~14% change-stability decrease**, explained via a J-curve (dip during platform build-out, recovery later). (Figures from the 2024 report as cross-reported by multiple independent summaries; the writer should verify exact numbers against the PDF.) **This is direct evidence that PR/delivery metrics mis-score platform work in-flight** — the brief's near-zero-scoring agent engineers are the individual-level version of this J-curve.
- Vendor metric catalogs (Jellyfish "17 platform engineering metrics," Datadog, platformengineering.org): converge on adoption rate, time-to-first-X, provisioning time, toil/KTLO allocation trend, developer-satisfaction surveys, and $-ROI = (reclaimed dev time × rate + infra savings − platform cost) / cost. SEO-grade numbers circulating in this space ("185–220% IDP ROI," "2.5x more features") are **unsourced marketing — excluded**.

### 1d. Automation impact measurement (SRE toil + RPA)
- **Google SRE Workbook, "Eliminating Toil"**: toil = manual, repetitive, automatable, reactive, no enduring value, scales with service growth. Measurement method: identify toil from the people doing it; pick an **objective, consistent, well-understood unit** (hours, tickets, manual production changes); measure continuously before/during/after automation; keep toil <50% of team time. Quantify automation ROI as time-saved vs. build+maintain cost, but Google explicitly lists benefits that resist hour-accounting (morale, fewer context switches, fewer human-error outages) and warns that "experience and intuition are not repeatable, objective, or transferable."
- **RPA / UiPath Automation Hub "Detailed Assessment Algorithm"** — the most concrete real-world implementation of per-automation value scoring found anywhere: **hours saved/yr = Automation Potential % × total as-is process hours/yr; $ benefit = Automation Potential % × (avg employee full cost × FTEs required)**. Every load-bearing input (process volume, time per transaction, rework/audit hours, employee cost, headcount affected) is **self-reported by the submitter and not independently validated**. This is the cautionary tale for any "hours saved" signal.
- RPA practitioner literature adds: hours-saved captures a minority of real value (cycle time, error reduction, audit posture, 24/7 coverage are missed), and programs justified purely on hours-saved either underinvest or overpromise.

### 1e. Non-PR knowledge-work measurement
- **"Invisible labor" research (arXiv 2401.06889 and successors)**: automated contribution tooling (GitHub graphs) counts only code, structurally down-ranking moderation, triage, docs, infra, governance — and this information "is rarely recorded in a standardized way" in repos. **CHAOSS** (Linux Foundation) maintains implementation-agnostic community-health metrics precisely because code-only counting fails; even there, non-code contribution measurement remains survey/self-report/manual-attribution heavy.
- **Tanya Reilly, "Being Glue"** [practitioner canon]: essential non-promotable work is invisible to metric systems and career ladders; unmanaged, it pushes people out of technical tracks. Directly names the fairness failure our brief's agent engineers are experiencing.
- Net: this literature diagnoses the invisibility problem thoroughly but offers **no telemetry-based per-individual solution**. Honest gap.

## 3. Known failure modes

1. **Runs/sessions/tokens are engagement, not value.** The entire standard telemetry stack (OTel, Langfuse, LangSmith) stops at cost/latency/errors. Cost-per-token in isolation is explicitly called a vanity metric (Google Cloud). A run-count-based subscore reproduces the current metric's flaw one level up.
2. **Success is builder-defined.** Langfuse scores, LLM-as-judge evals, and "resolution rate" definitions are all controlled by the person being measured. No neutral success oracle exists.
3. **Self-reported time-saved inflates.** UiPath's formula is driven end-to-end by unvalidated submitter inputs; RPA practice shows chronic over/under-statement. Google SRE concedes hour-accounting misses most indirect value.
4. **Platform value shows up in *other people's* metrics, with a lag (J-curve).** DORA 2024: platform investment depresses throughput/stability before it pays off. Any short-window individual metric will punish platform work while it is being built.
5. **Adoption metrics without retention reward launches, not value.** Spotify/Google Cloud both push past raw usage to repeat use, retention, and implicit rejection (reverts) for exactly this reason.
6. **Correlation ≠ causation in every platform case study.** Heavy platform users being more productive (Spotify 2.3x GitHub activity) is confounded by selection.
7. **Non-PR work is structurally unrecorded.** Invisible-labor research: the data does not exist in standard systems; no amount of metric cleverness conjures it.

## 4. Gameability & who it penalizes (per metric family)

All entries: (a) how gamed, (b) who unfairly penalized. Attack analysis is my inference unless cited.

- **Run counts / sessions / active days (HAVE today).** (a) Cron-schedule agents, self-invoke in loops, split one workflow into many runs; an agent that retries in a loop "earns" more. Zero marginal cost to inflate. (b) Builders of few high-leverage agents vs. many trivial ones; engineers whose agent runs headless under a service account (runs may not attribute to them at all).
- **Distinct other-user consumers (NEW signal).** (a) Astroturf: ask teammates to invoke it once; reciprocal run-my-agent rings; demo-driven spikes. Materially harder than self-inflation because it requires other humans' actions. (b) Engineers whose automation's "consumer" is a *system* (CI service account, scheduler) not a person; single-consumer but business-critical automations; new agents in cold start; niche-audience tools.
- **Repeat use / retention by others (e.g., consumers active in ≥3 distinct weeks).** (a) Hardest to game of the family — requires sustained voluntary behavior by others; still gameable via a small pact of colleagues, or by embedding your agent as a mandatory step in a workflow others can't avoid (which may actually be value, or may be rent-seeking). (b) Seasonal/quarterly tools; internal tools for small teams (low N looks like low value); deprecation of a superseded agent looks like decline.
- **Run success/error rate (cheap NEW signal: terminal status).** (a) Define success trivially; catch-and-suppress errors; kill doomed runs early (survivorship); the CI-clean-rate pathology repeats — iterate until green, distribution pins near 100%. (b) Builders attempting hard, novel tasks (same difficulty-penalty as per-dollar metrics); agents operating in flaky external environments.
- **Builder-defined eval scores (Langfuse-style).** (a) Owner writes the judge; LLM-as-judge is promptable to leniency. (b) Rigorous self-evaluators score worse than lenient ones — inverts the incentive the dashboard exists to create.
- **Self-reported hours saved (UiPath-style).** (a) Inflate baseline time/volume/headcount — UiPath's own docs concede inputs aren't validated. (b) Honest reporters; people automating small-but-frequent toil vs. impressive-sounding rare tasks.
- **Plugin/agent contribution counts (Backstage-style ecosystem metrics).** (a) Split one agent into five; publish trivial variants. (b) Maintainers who consolidate/deprecate; people who improve others' agents rather than authoring new ones (the glue-work trap again).
- **Toil-allocation trend / KTLO ratio.** (a) Reclassify work categories; survey self-report drift. (b) On-call-heavy roles (SRE) whose toil is externally imposed; team-level by nature, can't credit an individual.
- **Time-to-X org metrics (Spotify time-to-10th-PR).** (a) Individual can barely game an org metric — but also can't be credited by it. (b) Not usable per-person; listed to exclude.

## 5. Applicability to our constraints (mapped to HAVE / CANNOT)

**With HAVE data only** (per-user/day/tool: $, tokens, sessions, active days; assumed run counts; PRs/CI/cohorts):
- Nothing in this literature turns spend+runs into *value*. The only honest HAVE-only agent-platform subscore is an **engagement/consistency signal** (active building days, session regularity, breadth across tools) — a participation proxy, and it must be labeled as such on the dashboard, not dressed up as leverage. Anything stronger claimed from HAVE-only data would be false precision.
- One partial HAVE-only mitigation: **stop dividing agent-platform engineers' spend into a PR-denominated ratio.** Function-aware cohorts (in scope per brief) plus classifying users into "PR-output profile" vs. "agent-platform profile" by spend mix fixes the near-zero pathology *defensively* (they stop being punished) even before any new signal lands. This is my inference, but it follows directly from DORA 2024's J-curve evidence that delivery metrics mis-price platform work.

**Highest-lift NEW signals (priority order), each grounded in this survey:**
1. **Per-run: agent_id, owner_id, caller identity (human vs. service account), timestamp.** Enables the whole adoption/reuse family: distinct non-self human consumers, repeat-consumer retention, non-self run share. Every observability stack in this space (OTel semconv, Langfuse users/sessions) already models user identity on traces — the data almost certainly exists in platform logs today (inference). This single join converts the agent platform from a pure cost center into an output surface. Privacy-compatible: show the builder aggregate counts, never consumer names.
2. **Per-run terminal status (completed/errored/cancelled).** Cheap; weak alone (see gaming above); useful mainly to discount obviously-broken run spam, not as a quality score.
3. **Revert/undo-style implicit rejection where agents act on systems** (e.g., agent-opened PRs: merged vs. closed-unmerged) — Google Cloud's strongest adoption signal; only applies to the subset of agents that emit artifacts into systems we already observe (GitHub). Partially inside HAVE if agent-authored PRs can be linked to the owning engineer — the brief says attribution is unreliable, so treat as NEW.
4. **Builder-defined eval scores** (Langfuse-pattern): valuable for the builder's own self-reflection view; **never** comparable across people (owner-controlled). Display-only, uncohorted.
5. **Structured self-reported time-saved with caps and mandatory assumptions shown** (UiPath pattern, minus its validated inflation channels): calibration/reflection only, excluded from any score.

**Constraint fit:** self-view-only privacy is *easier* here than for leaderboards — adoption-by-others counts shown only to the builder are aggregate and anonymous by construction. Data lag (~2 weeks) is tolerable because reuse/retention metrics want 8–12-week windows anyway (retention math needs multiple weeks; also smooths the DORA J-curve). Cold start: any adoption metric needs an explicit "too new to score" state for agents <4 weeks old — a designed state, not a brittle floor (lesson from the brief's floor critique).

**Dead ends (honest):** (1) No way, under CANNOT constraints, to value agents whose consumers are machines/pipelines rather than people — flag as unmeasured, don't fake it. (2) No neutral per-run success oracle exists or is on the standards roadmap (OTel captures errors, not success). (3) Business value of an automation (Google Cloud pillar 3) requires per-workflow baselines we will never have per-engineer. (4) Glue/invisible work beyond the agent platform (mentoring, reviews, incident response) stays invisible; this survey found diagnosis, not cures.
